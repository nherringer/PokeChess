Overview
This document covers the HTTP contract between the app server and the engine
(ML bot) container, and the serialization layer that underlies it.

For deployment topology, S3 persistence, and ops endpoints (backup), see
docs/app_and_engine_communication.md.  For the full player-facing REST API,
see docs/api_spec.md.

Piece.to_dict() / Piece.from_dict(d)
All fields serialize directly; enum values become their .name string

stored_piece serializes recursively (nested dict or null)

id is included as a string UUID or null; the engine treats it as an opaque
carrier — it is set by the app layer and passed through unchanged.

ForesightEffect.to_dict() / ForesightEffect.from_dict(d)
Straightforward — all integer fields, no nesting

GameState.to_dict() / GameState.from_dict(d)
board → flat list of piece dicts (only occupied squares; row/col encode position)

active_player → team name string ("RED" or "BLUE")

pending_foresight → {"RED": null | <foresight dict>, "BLUE": null | <foresight dict>}

has_traded, foresight_used_last_turn → {"RED": bool, "BLUE": bool}

Move.to_dict() / Move.from_dict(d)
action_type → ActionType.name string (e.g. "MOVE", "ATTACK")

All optional fields (secondary_row, secondary_col, move_slot) included as null when absent

Endpoint: POST /move
This is the only endpoint the app calls in normal game operation.
(Ops/backup endpoints are separate — see docs/app_and_engine_communication.md.)

The app sends the current board state together with a persona_params dict that
controls MCTS behaviour.  The engine picks the best move and returns it.

Request body:
{
  "state":         { <FEN dict> },
  "persona_params": {
    "time_budget":      <float, seconds — MCTS wall-clock budget>,
    "exploration_c":    <float, optional — UCB1 exploration constant>,
    "rollout_depth":    <int,   optional — max rollout depth>
  }
}

  - "state"          The full current GameState serialized as a FEN dict (see Phase 1).
  - "persona_params" A dict of MCTS tuning knobs.  Only "time_budget" is required;
                     unrecognised keys must be silently ignored so future params can
                     be added without breaking old engine versions.
  - "time_budget"    Wall-clock seconds the engine may spend on MCTS search.
                     The app clamps this to [0.1, 30.0] before sending.

Response 200:
{
  "move": { <Move dict> }
}

Processing steps:
1. Deserialize state via GameState.from_dict()
2. Read persona_params["time_budget"] (required) and any other recognised knobs
3. Call mcts.select_move(state) with the given budget → Move
4. Return {"move": chosen_move.to_dict()}

Note on load-aware budget scaling
The app server computes an *effective* time_budget before calling this endpoint
(base_budget / N_active_players).  This is entirely app-side logic — the engine
simply honours whatever time_budget it receives in persona_params.
See docs/load_aware_budgeting.md for the full design.

Note on session / tree reuse
The engine maintains an in-process MCTS session keyed by game UUID so the search
tree is reused across turns.  Session management is internal to the engine and
not part of this contract.
