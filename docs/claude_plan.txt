The plan is written to the plan file. Here's a quick summary of what it covers:

  6 implementation groups, sequenced so each unblocks the next:

  1. Foundation — config, DB pool, main.py, Alembic migration (full 8-table schema)
  2. Auth + Users — register/login/refresh, JWT deps, GET /me, PATCH /me/settings
  3. Friends + Invites — PvP social layer, invite→pending game→active game flow
  4. Game Creation + Serialization — the core: state serialization, piece UUID tracking via parallel id_map (works around ML-engineer
  blockers), PvB game creation, game listing/viewing, resign
  5. Legal Moves + Engine Client — legal move filtering, httpx client for engine container
  6. Move Submission — the full lifecycle: apply move, handle pokeball RNG, detect foresight resolution, build history entries (10 action
  types), PvB bot response, terminal detection, XP attribution

  Key architectural decision: Piece UUIDs are tracked as a parallel dict[tuple[int,int], str|None] keyed by board position — completely
  independent of the engine's Piece dataclass. This sidesteps both ML-engineer blockers (Piece.id and to_dict/from_dict) so the app team can
  proceed immediately.

 PokeChess App Backend — Implementation Plan

 Context

 The app/ directory currently contains only a README.md. The branch init/app-backend is the scaffold commit. The engine (engine/) is complete
  and tested. Two ML-engineer blockers exist — Piece.id and GameState.to_dict()/from_dict() — but the plan below works around both so the app
  team can proceed independently.

 This plan produces a fully structured, runnable FastAPI backend matching the specs in docs/api_spec.md, docs/pokechess_data_model.md, and
 docs/implementation_roadmap.md.

 2. Dependency & Middleware Stack

 Dockerfile.app changes

 Add httpx, sqlalchemy, psycopg2-binary to the pip install line:
 RUN pip install --no-cache-dir \
     fastapi uvicorn[standard] asyncpg alembic \
     "passlib[bcrypt]" "python-jose[cryptography]" \
     httpx sqlalchemy psycopg2-binary
 (sqlalchemy + psycopg2-binary are Alembic-only; app code uses raw asyncpg.)

 docker-compose.yml changes

 Add to pokechess-app.environment:
 SECRET_KEY: "dev-secret-change-me"
 ACCESS_TOKEN_EXPIRE_MINUTES: "30"

 main.py lifespan

 @asynccontextmanager
 async def lifespan(app: FastAPI):
     app.state.db_pool = await asyncpg.create_pool(dsn)
     app.state.engine_client = httpx.AsyncClient(
         base_url=settings.ENGINE_URL, timeout=15.0
     )
     yield
     await app.state.engine_client.aclose()
     await app.state.db_pool.close()

 Middleware

 - CORSMiddleware (allow all origins in dev)
 - No rate limiting or request logging for v1

 Auth

 - passlib.hash.bcrypt for passwords
 - python-jose for JWT (HS256)
 - Access token: 30 min, returned in response body
 - Refresh token: 30 day, httpOnly cookie
 - get_current_user dependency: extracts Authorization: Bearer, decodes JWT, returns user row dict

 Error handling

 Custom AppError(status_code, error, detail) + exception handler → {"error": "...", "detail": "..."}

 3. Implementation Order

 Group 1: Foundation

 Goal: App starts, connects to Postgres, serves health check, migrations run.

 Files:
 - app/__init__.py
 - app/config.py
 - app/main.py (lifespan + CORS + error handler + health route)
 - app/db/__init__.py, app/db/connection.py
 - alembic.ini + app/db/migrations/env.py + script.py.mako
 - app/db/migrations/versions/0001_initial_schema.py — all 8 tables, all indexes, all constraints, seed bots row for "Metallic"

 Key decisions:
 - config.py: plain module-level vars via os.environ / os.environ.get(). No pydantic-settings (not installed, not needed).
 - DATABASE_URL arrives as postgresql+asyncpg://...; strip +asyncpg for raw asyncpg DSN. Strip both +asyncpg and swap to psycopg2 prefix for
 Alembic.
 - Alembic migrations use raw SQL inside op.execute() — no ORM models.

 Risks:
 - Must add sqlalchemy + psycopg2-binary to Dockerfile for Alembic.

 ---
 Group 2: Auth + User Endpoints

 Goal: Users can register, log in, refresh tokens, view profile.

 Files:
 - app/auth.py
 - app/schemas.py (start with auth + user schemas; extend later)
 - app/routes/__init__.py, app/routes/auth.py, app/routes/users.py
 - app/db/queries/__init__.py, app/db/queries/users.py, app/db/queries/settings.py

 Key decisions:
 - POST /auth/register also creates user_settings row (defaults).
 - GET /me joins users + pokemon_pieces for roster. Roster may be empty.
 - PATCH /me/settings uses COALESCE($N, column) for partial updates.

 ---
 Group 3: Friends + Invites

 Goal: PvP social layer — friend requests, game invites, pending→active transition.

 Files:
 - app/routes/friends.py, app/routes/invites.py
 - app/db/queries/friends.py, app/db/queries/invites.py

 Key decisions:
 - Friendship rows: always sort UUIDs so user_a_id < user_b_id. initiator_id tracks sender.
 - GET /friends: single query with CASE/UNION for friends, incoming, outgoing.
 - POST /game-invites: creates game_invites + games row (status='pending', state=NULL) in one transaction. Inviter = Red.
 - PUT /game-invites/{id} (accept): initializes game — builds state, injects UUIDs, sets status='active'. Creates rosters for either player
 if missing.
 - Catch asyncpg UniqueViolationError on one_active_pvp index → return 409.

 Risks:
 - Invite acceptance is the most transactional flow: invites update + games update + game_pokemon_map inserts + possibly pokemon_pieces
 inserts. Must be one DB transaction.

 ---
 Group 4: Game Creation + Serialization + Read Endpoints

 Goal: PvB game creation works. State serialization round-trips. Games are listable/viewable.

 Files:
 - app/routes/games.py
 - app/db/queries/games.py, app/db/queries/pieces.py, app/db/queries/game_map.py
 - app/game_logic/__init__.py
 - app/game_logic/serialization.py
 - app/game_logic/id_map.py
 - app/game_logic/roster.py

 Key decisions:

 Piece UUID strategy (ML-engineer blocker workaround):
 The engine's Piece dataclass has no id field and apply_move() copies via dataclasses.replace() which would drop any monkey-patched
 attribute. Solution: maintain piece UUIDs as a parallel id_map: dict[tuple[int,int], str|None] keyed by board position. The engine never
 sees IDs. The app manages them:
 - Deserialization (state_from_dict): parse JSONB board[], extract each piece's id into id_map, construct Piece objects without id.
 - Serialization (state_to_dict): for each board piece, look up id_map.get((piece.row, piece.col)) and include it in the output JSON.
 - After apply_move: remap id_map based on what happened (tracked piece moved, captured, etc.). All other pieces are positionally stable.

 This completely decouples the app from the Piece.id blocker. When the ML engineer adds the field, the app can optionally migrate, but this
 approach continues to work.

 Serialization (serialization.py):
 - state_to_dict(state, id_map) → dict — produces the exact JSONB shape from pokechess_data_model.md
 - state_from_dict(data) → (GameState, id_map) — reconstructs GameState + extracts id_map
 - ~100 lines. Covers board[], pending_foresight, has_traded, foresight_used_last_turn, active_player, turn_number, stored_piece (recursive)

 ID remapping (id_map.py):
 - remap_ids(old_state, new_state, move, id_map) → new_id_map
 - Handles every action type: MOVE (piece relocates), ATTACK (attacker may move to target square on KO), QUICK_ATTACK (attack then secondary
 move), pokeball (both removed or attacker removed), RELEASE (safetyball consumed, stored piece appears), FORESIGHT (no position change),
 EVOLVE (no position change), TRADE (no position change)
 - Must also handle: foresight resolution (piece killed at turn start), safetyball discharge (stored pieces auto-released when player uses
 another piece), safetyball store (ally absorbed on MOVE), pawn promotion (position same, type changes)
 - This is the trickiest code in the app. Needs thorough testing.

 Roster creation (roster.py):
 - Creates 8 pokemon_pieces rows: 1 king, 1 queen, 2 rooks, 2 knights, 2 bishops
 - King species depends on player_side: "pikachu" (red) or "eevee" (blue)
 - Returns a position→UUID map for injecting into the starting board

 Game creation (POST /games PvB):
 1. Validate bot_id exists in bots table
 2. Create user's roster if first game (SELECT EXISTS FROM pokemon_pieces WHERE owner_id = $1)
 3. Build GameState.new_game()
 4. Fetch roster, build id_map by mapping role+position to UUID
 5. Serialize state + id_map → JSONB
 6. Insert games row (status='active', whose_turn='red', etc.)
 7. Insert game_pokemon_map rows for human's pieces
 8. Return GameDetail

 GET /games/{id}: Fetch full row, verify participant, return GameDetail.
 GET /games: Two queries (active + last 10 completed). Return GameSummary objects (no JSONB).
 POST /games/{id}/resign: Set status/winner/end_reason, trigger XP processing.

 ---
 Group 5: Legal Moves + Engine Client

 Goal: Frontend can query legal moves. Engine client ready for PvB.

 Files:
 - app/routes/moves.py (legal_moves portion)
 - app/engine_client.py

 Key decisions:
 - GET /games/{id}/legal_moves?piece_row=N&piece_col=N: deserialize state, call get_legal_moves(state), filter by piece position. Return Move
  fields with action_type as uppercase enum name string.
 - Validate: it's the player's turn, piece exists at given square, piece belongs to active player.
 - engine_client.py: httpx.AsyncClient (from lifespan's app.state). POST to /move, timeout = time_budget + 5s. Raise AppError(503) on
 failure.

 ---
 Group 6: Move Submission (most complex endpoint)

 Goal: Full move lifecycle — human move + PvB bot response + history + terminal + XP.

 Files:
 - app/routes/moves.py (POST /games/{id}/move)
 - app/game_logic/history.py
 - app/game_logic/xp.py

 POST /games/{id}/move flow:
 1. Auth + turn validation
 2. Fetch game row (state JSONB + move_history + bot fields)
 3. state, id_map = state_from_dict(row['state'])
 4. Parse request → Move object. Validate move in get_legal_moves(state).
 5. Snapshot pre-move pieces for history diff
 6. outcomes = apply_move(state, move); resolve pokeball RNG
 7. id_map = remap_ids(old_state, new_state, move, id_map)
 8. Detect foresight resolution → build foresight_resolve entry if so
 9. Build the move's history entry (10 action types, each with specific result fields)
 10. done, winner = is_terminal(new_state)
 11. PvB, not done: serialize state → call engine → parse bot move → apply → remap ids → detect foresight → build history → check terminal
 12. Single UPDATE games SET state=$1, move_history = move_history || $2::jsonb, ...
 13. On game over: compute_xp() → update game_pokemon_map + pokemon_pieces
 14. Return GameDetail

 History builder (history.py):

 Maps engine ActionType + piece context → history action_type:

 ┌───────────────────┬─────────────────┬─────────────────────┐
 │ Engine ActionType │  Piece context  │ History action_type │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ MOVE              │ any             │ move                │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ ATTACK            │ POKEBALL        │ pokeball_attack     │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ ATTACK            │ MASTERBALL      │ masterball_attack   │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ ATTACK            │ named piece     │ attack              │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ QUICK_ATTACK      │ Eevee           │ quick_attack        │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ FORESIGHT         │ Mew/Espeon      │ foresight           │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ TRADE             │ any             │ trade               │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ EVOLVE            │ any             │ evolve              │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ RELEASE           │ Safetyball      │ release             │
 ├───────────────────┼─────────────────┼─────────────────────┤
 │ (auto)            │ foresight fires │ foresight_resolve   │
 └───────────────────┴─────────────────┴─────────────────────┘

 Each type produces a different result dict. The builder computes damage, type_multiplier, HP before/after, captured status, etc. by diffing
 old vs new state. For attacks, it uses engine.rules._calc_damage logic (or re-derives from the MATCHUP table).

 Foresight resolution detection:
 Compare old_state.pending_foresight[active_player] before vs after apply_move. If it was set with resolves_on_turn == old_state.turn_number
 and is now None, foresight resolved. Build the history entry from the foresight effect + target piece HP changes.

 Note: apply_move internally calls _resolve_foresight at the START of the turn, before executing the player's chosen move. So foresight
 resolution happens inside apply_move — the app detects it by comparing states, not by doing it separately.

 XP computation (xp.py):
 def compute_xp(move_history: list[dict]) -> dict[str, int]:
     xp = {}
     for entry in move_history:
         pid = entry.get("piece_id")
         if pid is None:
             continue
         damage = entry.get("result", {}).get("damage")
         if damage and damage > 0:
             xp[pid] = xp.get(pid, 0) + damage
     return xp

 Risks:
 - History builder is the most error-prone code. Each of 10 action types needs exact field coverage.
 - PvB double-move can produce up to 4 history entries (human foresight_resolve + human move + bot foresight_resolve + bot move).
 - Trade is a free action (doesn't end turn) unless Eevee auto-evolves — whose_turn stays same when trade doesn't end turn.
